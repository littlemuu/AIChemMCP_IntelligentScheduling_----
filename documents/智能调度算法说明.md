# 智能调度算法说明

## 1. 问题定义

该调度系统面向自动机器人实验室，系统中包含两类核心资源：

- 工作站：负责对样品执行实验步骤。
- 机器人：负责在工作站之间转运样品。

每个任务都包含一条固定的实验流程，即样品需要依次经过若干工作站。多个任务可能并发执行，并且可能竞争同一批工作站和机器人资源。

系统的目标不是单纯“有空就上”，而是在满足资源约束的前提下，让任务稳定推进，并处理一类特殊约束：**无缝连接步骤**。

所谓无缝连接，是指：

- 当前步骤完成后，不能在原工作站等待；
- 必须立即由机器人接走；
- 目标工作站在样品到达时必须已经可用；
- 不能在目标工作站前排队。


## 2. 核心思路

该系统适合采用一种**事件驱动 + 前瞻预留**的调度算法。

核心思想如下：

1. 当工作站空闲时，调度器决定是否让它开始处理一个新任务。
2. 在决定之前，不只看“当前能不能开始”，还要向前检查后续关键资源是否能衔接上。
3. 如果后续资源也能对齐，则提前把这些资源的未来时间窗口一起预留。
4. 当工作站真正完成时，机器人不再临时决策，而是直接执行预留好的转运动作。

这种做法的优势是：

- 能处理普通任务和无缝任务；
- 能减少“做完后卡住”的情况；
- 能把工作站调度和机器人调度统一到同一套时间轴模型中。


## 3. 数据模型

### 3.1 任务模型 `Task / Job`

任务至少需要维护以下字段：

- `TaskID`：任务唯一标识
- `Workflow`：有序工作站列表，例如 `[W1, W3, W2]`
- `WorkflowTools`：每一步需要的工具列表，例如 `[T2, T3, T1]`
- `ProcessingTimes`：各步骤处理时长
- `SeamlessSteps`：需要无缝连接的步骤对，例如 `[(0, 1)]`
- `Status`：任务状态，例如 `WAITING / RUNNING / COMPLETED`
- `CurrentStep`：当前执行到第几步
- `SampleID`：样品标识

### 3.2 资源模型 `Resource`

机器人和工作站都可以抽象为资源，公共字段包括：

- `ID`：资源唯一标识
- `Status`：资源状态
- `Timeline`：未来预留时间轴
- `CurrentTaskId`：当前关联任务

建议资源状态至少包含：

- `IDLE`：空闲
- `BUSY`：正在处理
- `COMPLETED` 或 `COMPLETED_WAITING_FOR_PICKUP`：已完成，等待转运
- `RESERVED`：未来时间段已被预留
- 机器人还可包含：`MOVING_TO_PICKUP`、`TRANSPORTING`

### 3.3 机器人模型 `Robot`

机器人可以在资源基类之上维护：

- 当前运输任务
- 取样耗时
- 运输耗时
- 可选：当前位置、移动时间矩阵

### 3.4 工作站模型 `Workstation`

工作站可以在资源基类之上维护：

- `Tools`：该工作站具备的工具集合

### 3.5 工具模型 `Tool`

- `id`：工具唯一标识


## 4. 调度触发事件

算法围绕两个关键事件运行。

### 4.1 事件 A：工作站变为 `IDLE`

这是主调度入口，也是主要决策点。

当某个工作站空闲时，调度器需要判断：

- 是否要让它开始一个新任务；
- 或者是否要承接某个正在执行任务的下一步；
- 是否需要为后续无缝步骤提前预留机器人和目标工作站。

### 4.2 事件 B：工作站变为 `COMPLETED`

这是机器人调度的执行点。

如果该步骤的转运已经提前预留，那么此时不再重新做选择，只需要：

- 找到被预留的机器人；
- 下发取样和运输指令；
- 更新工作站和机器人状态。


## 5. 事件 A 的详细流程

### 5.1 收集候选任务

当工作站 `W_i` 变为空闲时，候选来源主要有两类：

- 新任务：任务处于 `WAITING`，且 `W_i` 是它流程的第一站；
- 后续任务：任务已经启动，`W_i` 是该任务下一步目标站。

### 5.2 可行性检查

对每一个候选任务，不应直接下发执行，而应先做一次前瞻检查。

以流程 `W_i -> W_j` 为例：

1. 计算当前步骤完成时间  
   `T_finish_i = now + processing_time(W_i)`

2. 查找可用机器人  
   找到一个机器人 `R_k`，使其能在 `T_finish_i` 前后接手该样品。

3. 计算转运耗时  
   包括取样、搬运、放样等动作的耗时。

4. 计算样品到达下一工作站时间  
   `T_arrival_j = T_finish_i + T_transport`

5. 检查目标工作站可用性  
   判断 `W_j` 在 `[T_arrival_j, T_arrival_j + processing_time(W_j))` 内是否空闲。

6. 若三者都可满足  
   当前工作站、机器人、目标工作站三者都可对齐，则该候选可行。

### 5.3 无缝连接的处理

如果 `(step_i, step_j)` 被标记为无缝连接，则必须一次性同时满足：

- 当前工作站可按时完成；
- 机器人能在完成时立刻接手；
- 目标工作站在到达时立刻可开始处理。

任意一个条件不满足，都不能启动该任务的当前步骤。

这正是前瞻预留机制的核心价值。

### 5.4 多候选选择策略

如果多个候选都可行，则需要优先级规则。常见策略如下：

- `FCFS`：先来先服务，优先等待时间最长的任务
- `SPT`：优先当前步骤耗时最短的任务
- `Task Priority`：按任务业务优先级处理
- `Global Optimization`：以总完工时间或吞吐量为目标进行全局优化

对实时实验室场景，建议先使用：

- 带优先级的 `FCFS`
- 或者简单的 `SPT`

实现成本低，也更容易调试。

### 5.5 正式预留与下发命令

当选出最优候选后，应将模拟结果转为正式预留：

- 将当前工作站写入 `timeline`
- 将机器人未来转运区间写入 `timeline`
- 将下一工作站未来处理区间写入 `timeline`
- 更新相关资源状态为 `BUSY` 或 `RESERVED`
- 向硬件发送开始处理命令


## 6. 事件 B 的详细流程

当工作站 `W_i` 状态变为完成时：

1. 查询该任务是否已有预留的机器人与目标工作站；
2. 若有，则直接向机器人下发取样和转运命令；
3. 更新机器人状态为 `MOVING_TO_PICKUP` 或 `TRANSPORTING`；
4. 将源工作站从 `COMPLETED_WAITING_FOR_PICKUP` 释放为 `IDLE`；
5. 样品送达后推进任务的 `CurrentStep`；
6. 若下一步未提前预留，则将任务重新放回候选队列。

因此，事件 B 更接近“执行已定计划”，而不是“重新做调度决策”。


## 7. 时间轴模型

时间轴 `Timeline` 是整个算法的核心结构。

它的作用是：

- 记录资源未来的占用和预留情况；
- 支持可行性检查；
- 支持无缝连接的前瞻判断；
- 避免资源在未来时间段冲突。

示例：

```text
W2.timeline = [
  (T1, 10:00, 10:15),
  (T3, 10:30, 10:50)
]
```

表示该工作站在两个未来时间段已被占用或预留。

调度器判断资源是否可用时，只需检查候选时间区间是否与 `timeline` 中已有区间重叠。


## 8. 安全机制

### 8.1 时间戳

所有事件和状态更新建议都携带精确时间戳。

原因是：

- 通信可能延迟；
- 事件到达顺序可能与实际发生顺序不一致；
- 调度器应根据“事件发生时间”而不是“消息收到时间”做判断。

### 8.2 安全缓冲

所有处理时间和运输时间建议引入安全缓冲，例如：

- 固定增加 15 秒
- 或按比例增加 5%

这样可以降低硬件抖动、通信延迟、机械动作波动带来的计划失效风险。

### 8.3 指令确认闭环

如果系统通信链路不够稳定，建议加入 ACK 机制：

1. 调度器发送指令；
2. 硬件确认是否收到且是否可执行；
3. 返回 `ack_status: success`；
4. 调度器在收到 ACK 后，再将任务状态正式改为 `RUNNING`。

如果通信链路稳定且平台简单，这部分可以后续补充。


## 9. 推荐实现方式

从工程实现角度，建议把系统拆成以下几层：

- 模型层：`Task`、`Robot`、`Workstation`、`Tool`
- 调度层：候选生成、优先级排序、可行性检查、资源预留
- 状态推进层：根据时间轴和硬件事件更新资源状态
- 执行层：向硬件发送命令并接收回执

运行机制建议采用：

- 事件驱动为主
- 时间轴预留为核心
- 状态机推进为辅助


## 10. 总结

这套算法的本质是：

**通过“先模拟、再预留、后执行”的方式，把工作站调度和机器人调度统一起来，并通过对未来资源窗口的联合检查来保证无缝连接步骤的可执行性。**

它适合以下场景：

- 多工作站、多机器人并发
- 多任务共享资源
- 存在必须连续衔接的实验步骤
- 需要较强可解释性和可落地性

如果后续任务规模扩大，或希望进一步优化吞吐量，可以在当前框架上逐步引入：

- 更复杂的优先级策略
- 路径与位置建模
- 全局优化算法
- 异常恢复与重调度机制
