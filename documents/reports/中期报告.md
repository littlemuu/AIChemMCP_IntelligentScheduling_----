# 中期报告

## 1. 项目概述

本项目以自动化实验室场景为背景，围绕“上层实验任务编排”与“下层实验资源调度”两个核心问题展开研究与实现。项目的总体目标，是在现有 AIMCP 框架基础上，引入智能调度算法，使系统能够逐步具备任务接收、资源分配、状态推进和结果反馈等能力，从而形成面向自动化实验任务执行的原型系统。

从当前研究内容看，本项目不仅关注单个模块的实现，更关注不同模块之间的衔接问题。具体而言，AIMCP 更适合承担上层任务理解、工具调用和实验流程编排的职责，而智能调度模块则更适合承担工作站分配、机器人调度、时间窗预留和任务状态管理等职责。因此，本阶段的主要工作重点并不是单纯完成某一个独立功能，而是建立一个能够贯通上层和下层的初步系统结构。

中期阶段的工作目标可以概括为以下几点。

(1) 梳理项目原始资料与原始代码结构，明确系统边界。

(2) 分析并整理智能调度算法的核心逻辑，形成可解释的调度框架。

(3) 设计 AIMCP 与调度模块之间的连接方式，并在代码层实现最小闭环。

(4) 形成可运行的原型，以支持后续功能扩展、测试分析和课程汇报。


## 2. 需求分析

需求分析是系统设计的基础。本项目的需求既来自自动化实验室场景本身，也来自当前已有 AIMCP 框架与调度模块之间的耦合问题。结合现阶段代码基础、文档资料和预期目标，当前项目需求可分为背景需求、功能需求、非功能需求和阶段落实情况四个部分。

### 2.1 项目背景需求

自动化实验室场景中的实验任务，通常不是由单一设备独立完成，而是需要多个工作站、实验工具和机器人协同参与。例如，一个样品可能需要先在某一工作站完成反应，再由机器人转运至另一工作站进行测量，之后还可能继续进入表征流程。在此过程中，任务之间会共享资源，工作站之间会发生竞争，机器人还需要承担转运与衔接任务。

另一方面，AIMCP 作为上层任务编排系统，更擅长根据目标调用工具、构造实验流程和组织执行逻辑，但并不直接解决底层物理资源如何分配的问题。因此，如果没有调度模块，上层生成的任务将难以映射为真实可执行的资源安排；如果只有调度模块而没有上层任务编排，系统又难以根据实验目标动态生成新任务。

因此，本项目在背景层面的核心需求是：构建一个同时覆盖“实验目标生成”和“实验资源执行”的中间系统，使上层 AIMCP 和下层实验调度在统一框架下协同工作。

### 2.2 功能需求

结合自动化实验平台的典型使用场景，当前项目需要满足如下功能需求。

(1) 系统应能够接收来自用户或上层智能体的实验任务请求，并将其解析为系统可处理的任务对象。

(2) 系统应能够描述实验任务的步骤链、处理时间、样品标识以及步骤之间的关系，从而支撑后续调度逻辑。

(3) 系统应能够维护工作站、机器人和任务的实时状态信息，并支持状态查询。

(4) 系统应能够在多工作站、多机器人、多任务并行的场景下进行基本调度。

(5) 系统应支持普通步骤与无缝衔接步骤两类实验流程，其中无缝衔接步骤应保证当前工作站、机器人和目标工作站在时间上可连续衔接。

(6) 系统应能够支持任务从提交、运行、等待转运到最终完成的全过程推进。

(7) 系统应能够返回调度结果和运行状态，以便上层系统继续决策或展示结果。

### 2.3 非功能需求

除业务功能之外，系统还需满足一定的工程性要求。

(1) 系统结构应尽量模块化，以便后续在不破坏总体结构的前提下继续扩展。

(2) 调度逻辑应具备较好的可解释性，便于课程汇报、问题定位和后续优化。

(3) AIMCP 模块和调度模块之间应保持相对清晰的边界，避免上层逻辑与底层执行耦合过深。

(4) 当前系统应支持从“本地模拟执行”逐步过渡到“真实硬件接口接入”。

(5) 项目应保留原始版本与集成版本，便于对照、回退和展示阶段性演化过程。

### 2.4 当前需求落实情况

从中期阶段的完成情况来看，当前已经初步落实了部分关键需求。

(1) 已经实现从 AIMCP 动作请求到调度任务对象的转换。

(2) 已经建立 `Task`、`Resource`、`Workstation`、`Robot`、`Tool` 等核心数据模型。

(3) 已经实现任务提交、状态查询、调度推进和运行至完成等基础能力。

(4) 已经完成 AIMCP 与调度模块之间的最小闭环连接。

(5) 已经形成一个可运行的原型，可用于展示系统基本流程。

与此同时，当前仍未完全满足的需求主要包括以下几方面。

(1) 真实实验设备接口尚未接入。

(2) 多步复杂实验链尚未形成稳定的完整演示。

(3) 调度器与执行层之间的回执闭环尚未建立。

(4) LLM 自动决策与调度联动仍处于原型阶段。


## 3. 功能模块框架与设计（概要设计）

概要设计的主要目标，是从系统整体层面说明项目采用了怎样的结构划分，各模块之间如何协同，以及当前架构为什么适合本项目场景。

### 3.1 总体设计思路

本项目当前采用分层设计思路，将系统划分为上层任务编排层、中层调度运行层和下层执行表现层。这样的设计具有以下几个优点。

(1) 可以避免上层智能体直接操作底层设备细节，降低耦合度。

(2) 可以将调度逻辑集中管理，便于统一处理资源冲突、状态推进和任务预留。

(3) 可以为后续真实硬件接入保留中间层接口，而不需要重写上层系统。

因此，从系统整体上看，本项目当前结构并不是将 AIMCP 与调度器“混合”在一起，而是将它们组织为“上层调用调度层、调度层管理执行层”的关系。

### 3.2 功能模块划分

当前系统可划分为以下几个主要模块。

(1) Agent 模块。

负责接收用户目标、发现可用工具、向对应服务端发起调用，并在后续承担上层任务编排角色。

(2) Action Server 模块。

负责接收动作类 MCP 请求，将请求路由到工具实现层，是 AIMCP 与调度模块连接的协议入口。

(3) ActionServerTools 模块。

负责将上层动作请求转换为调度运行时中的任务提交、状态查询和调度推进操作。

(4) SchedulingRuntime 模块。

负责维护调度器实例、系统当前时间、资源状态以及状态推进流程，是调度层的运行管理中心。

(5) Scheduler 模块。

负责执行核心调度逻辑，包括候选任务选择、资源可用性判断、时间窗预留和命令生成。

(6) 数据模型模块。

负责统一描述任务、工作站、机器人和工具等对象，是整个调度系统的数据基础。

### 3.4 模块关系说明

为了进一步说明模块之间的职责边界，当前系统采用如下关系。

(1) Agent 不直接操作工作站和机器人。

(2) Action Server 不直接做调度决策，而只负责通信和路由。

(3) ActionServerTools 不直接完成复杂调度，而是作为“请求翻译层”。

(4) SchedulingRuntime 负责长期维护状态，是调度执行过程的承载层。

(5) Scheduler 负责资源级决策，是当前项目的算法核心。

这一设计保证了系统的职责划分相对清晰，也为后续扩展提供了较好的基础。


## 4. 模块具体实现（详细设计）

详细设计部分主要说明当前已经实现的核心模块、内部数据结构以及它们在运行过程中承担的具体功能。

### 4.1 数据模型模块实现

当前已实现以下调度核心数据模型。

(1) `Task`

用于描述实验任务，主要包含任务编号、流程步骤、处理时间、样品标识、当前步骤和任务状态等信息。该对象是调度器处理的核心单元。

(2) `Resource`

作为工作站和机器人的统一抽象，维护资源编号、当前状态、时间线以及当前执行任务编号。通过统一抽象，调度器可以用类似方式管理工作站和机器人。

(3) `Workstation`

继承自 `Resource`，增加工具集合属性，用于表示实验工作站及其具备的处理能力。

(4) `Robot`

继承自 `Resource`，用于表示样品转运机器人。

(5) `Tool`

用于表示实验工具及其与工作站之间的映射关系，使系统能够根据步骤需求找到对应工作站。

### 4.3 调度器模块实现

调度器模块是当前系统的算法核心，已完成以下关键功能。

(1) 候选任务选择。

系统能够从任务队列中识别当前可在某个工作站上启动的任务。

(2) 资源可用性判断。

系统通过 `timeline` 机制判断未来时间窗是否存在冲突，从而确定当前调度方案是否可行。

(3) 资源预留。

在调度方案可行的前提下，系统会将任务写入时间线，对未来工作站和机器人资源进行预留。

(4) 无缝衔接判断。

对于存在连续衔接要求的步骤，系统会同时检查当前工作站、机器人和目标工作站的时间窗。

(5) 调度命令生成。

调度器当前可生成如 `START_PROCESSING` 等命令，用于表示资源执行动作。

### 4.4 调度运行时模块实现

`SchedulingRuntime` 是调度运行层的核心模块，负责维护调度器实例和整个系统的运行状态。当前已完成以下功能。

(1) 初始化工作站、机器人和工具映射。

(2) 接收来自 Action 层的任务提交请求。

(3) 推进系统时间并更新资源状态。

(4) 调用调度器进行调度并执行生成的命令。

(5) 提供状态查询接口和推进到任务完成的接口。

可以认为，`SchedulingRuntime` 是当前系统中的“调度运行容器”，它承接了算法逻辑和外部调用之间的联系。

### 4.5 Action 层模块实现

当前 `ActionServerTools` 已经从原始的空实现演进为具有实际功能的工具层。其核心作用是将 AIMCP 上层请求翻译为调度层操作。例如：

(1) 将 `robotic_reaction` 请求转化为反应任务对象并提交给调度运行时。

(2) 将 `robotic_measurement` 请求转化为测量任务对象并提交给调度运行时。

(3) 将 `robotic_characterization` 请求转化为表征任务对象并提交给调度运行时。

(4) 通过 `scheduler_status` 返回当前系统状态。

(5) 通过 `scheduler_advance` 与 `scheduler_run_until_complete` 驱动调度推进。

这一实现使 Action 层不再只是简单的空壳接口，而成为 AIMCP 与调度器之间真正的连接桥梁。

### 4.6 Agent 模块实现

当前 Agent 模块已经具备原型演示能力。其主要实现内容包括：

(1) 启动 BO Server 和 Action Server。

(2) 发现各 Server 所暴露的工具。

(3) 发送演示任务请求。

(4) 接收并展示调度返回结果。

(5) 在当前原型阶段，通过固定演示流程验证整个系统链路。

在中期阶段，Agent 已经具备顶层调用入口的基础能力，为后续引入更完整的 LLM 自动决策机制预留了条件。

## 5. 测试分析与案例分析

测试分析部分的主要目的是说明：当前系统是否已经具备最小闭环运行能力，以及当前测试是否能够支持中期阶段成果展示。

### 5.1 测试目标

本阶段测试主要围绕以下几个问题展开。

(1) AIMCP 与调度模块是否已经在代码层面打通。

(2) 任务是否能够成功从提交态进入运行态。

(3) 调度运行时是否能够将任务推进至完成态。

(4) 任务完成后资源是否能够被正确释放。

(5) 当前系统是否已经具备最小闭环展示能力。

### 5.2 测试方法

当前阶段主要采用本地演示方式进行测试。在集成版本目录下运行：

```powershell
python agent.py
```

该命令会自动执行一个最小闭环流程，包括以下步骤。

(1) 启动 BO Server 与 Action Server。

(2) 提交一个反应任务。

(3) 提交一个测量任务。

(4) 查询当前调度状态。

(5) 推进调度器直至任务完成。

### 5.3 测试结果分析

通过当前阶段实际运行，观察到了以下测试结果。

(1) Agent 能够成功启动 BO Server 与 Action Server。

(2) Agent 能够发现两个服务端所暴露的工具。

(3) 反应任务与测量任务能够被 Action Server 接收并提交给调度运行时。

(4) 调度器能够生成任务运行状态，并在推进后将任务状态更新为 `COMPLETED`。

(5) 工作站和机器人在任务完成后能够恢复为 `IDLE` 状态。

这些结果说明，当前系统已经基本满足中期阶段“形成可运行最小闭环原型”的目标。

### 5.5 案例分析

本阶段典型测试案例为一个简单的“反应 + 测量”演示流程。

(1) Agent 首先发起 `robotic_reaction` 请求，提交反应任务。

(2) 调度器将该任务分配给反应工作站，并将任务状态更新为运行中。

(3) Agent 随后发起 `robotic_measurement` 请求，提交测量任务。

(4) 调度器将该任务分配给测量工作站，并记录相应运行状态。

(5) 通过 `scheduler_run_until_complete` 推进运行时，系统持续更新任务与资源状态。

(6) 最终系统返回任务完成事件，并显示各工作站与机器人回到空闲状态。

该案例虽然较为简单，但已经能够体现当前系统的关键能力，即：上层任务请求能够被调度层接收、处理、推进并形成最终结果反馈。


## 6. 当前存在的问题

从中期阶段看，当前系统虽然已经完成初步集成并具备运行能力，但仍存在以下不足。

(1) 当前执行层仍以本地模拟为主，尚未真正接入实验设备控制接口。

(2) 多步复杂实验流程尚未形成稳定、完整的执行链路。

(3) 任务完成仍较依赖时间推进，尚未建立真实事件回执闭环。

(4) LLM 自动规划与调度联动仍处于原型验证阶段。
