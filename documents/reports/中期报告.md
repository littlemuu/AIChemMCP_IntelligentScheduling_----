# 中期报告

## 1. 项目概述

本项目以自动化实验室场景为背景，研究目标是将 AIMCP 上层任务编排能力与智能调度算法模块相结合，形成一个具有任务提交、资源调度、状态推进和结果反馈能力的实验平台。当前中期阶段的主要任务，是在完成原始资料梳理的基础上，建立项目的初步系统结构，并形成可运行的最小闭环。


## 2. 需求分析

### 2.1 项目背景需求

在自动化实验室环境中，实验任务的执行通常不仅涉及单个设备，而是需要多个工作站和机器人共同协作完成。与此同时，上层系统还需要根据实验目标不断生成新任务、调整任务顺序并跟踪任务状态。因此，本项目需要解决的并不是单纯的“设备控制”问题，而是“上层任务编排”与“下层资源调度”之间的协同问题。

### 2.2 功能需求

结合项目目标和当前场景，系统需要满足如下功能需求。

(1) 能够接收来自用户或上层智能体的实验任务请求。

(2) 能够将实验任务转换为调度器可处理的任务对象。

(3) 能够维护工作站、机器人和任务的状态信息。

(4) 能够在多任务、多资源场景下完成基本调度。

(5) 能够支持普通步骤与无缝衔接步骤两类实验流程。

(6) 能够查询运行状态，并支持任务从提交到完成的全过程跟踪。

### 2.3 非功能需求

除功能需求外，系统还应满足以下非功能需求。

(1) 系统结构应清晰，便于后续继续扩展。

(2) 调度逻辑应具备较强的可解释性，便于汇报和调试。

(3) 代码组织应具有模块化特点，便于 AIMCP 模块和调度模块解耦。

(4) 系统应能够从当前模拟执行模式平滑过渡到真实硬件接口模式。

### 2.4 当前需求落实情况

当前中期阶段已经初步满足以下需求。

(1) 已实现任务从 AIMCP 动作请求到调度任务对象的转换。

(2) 已建立任务、工作站、机器人、工具等核心数据模型。

(3) 已实现最小的任务提交、状态查询与运行推进能力。

(4) 已初步完成 AIMCP 与调度模块之间的对接。

尚未完全满足的需求主要集中在真实回执机制、多步复杂流程和真实硬件接口层面。


## 3. 功能模块框架与设计（概要设计）

### 3.1 总体设计思路

本项目当前采用分层设计思想，将系统划分为上层任务编排、中层调度运行和下层执行模拟三个层次。这样设计的目的，是避免上层智能体直接控制底层硬件，从而提高系统可维护性和扩展性。

### 3.2 总体模块划分

当前系统可划分为以下几个主要模块。

(1) Agent 模块

负责接收用户目标、发现工具并向对应 Server 发起调用。

(2) Action Server 模块

负责接收动作类 MCP 请求，并将其转发到工具实现层。

(3) ActionServerTools 模块

负责将上层动作请求转换为调度运行时可识别的任务提交或控制操作。

(4) SchedulingRuntime 模块

负责维护调度器实例、时间推进和状态更新，是当前调度层的运行管理中心。

(5) Scheduler 模块

负责执行核心调度逻辑，包括资源预留、可行性判断和命令生成。

(6) 数据模型模块

包括 `Task`、`Resource`、`Workstation`、`Robot`、`Tool` 等，用于统一描述调度对象。

## 4. 模块具体实现（详细设计）

### 4.1 数据模型模块实现

当前已实现以下调度核心数据模型。

(1) `Task`

用于描述实验任务，包括任务编号、步骤链、处理时间、样品标识、状态和当前步骤等信息。

(2) `Resource`

作为工作站和机器人的统一抽象，维护资源状态、时间线和当前任务编号。

(3) `Workstation`

继承自 `Resource`，增加工具集合属性，用于表示实验工作站。

(4) `Robot`

继承自 `Resource`，用于表示样品转运机器人。

(5) `Tool`

用于表示实验工具及其与工作站之间的映射关系。

### 4.2 调度器模块实现

调度器模块当前已完成以下关键逻辑。

(1) 候选任务选择。

系统能够从任务队列中识别当前可进入某工作站执行的任务。

(2) 资源可用性判断。

通过 `timeline` 机制检查工作站和机器人在未来时间窗内是否可用。

(3) 资源预留。

在确定可行后，调度器会将任务写入时间线，实现未来时段的资源占用预留。

(4) 无缝衔接判断。

对于要求连续衔接的步骤，调度器会同时检查当前工作站、机器人和下一工作站的可行性。

(5) 命令生成。

调度器会生成 `START_PROCESSING` 等调度命令，用于驱动后续执行流程。

### 4.3 调度运行时模块实现

在当前工程中，`SchedulingRuntime` 负责调度器实例管理和运行时状态维护，主要完成以下功能。

(1) 初始化工作站、机器人和工具映射。

(2) 接收来自 Action 层的任务提交请求。

(3) 推进时间并更新资源状态。

(4) 调用调度器生成新的调度命令。

(5) 支持运行到所有任务完成。

### 4.4 Action 层模块实现

`ActionServerTools` 当前已经从原始的空实现演进为具有实际功能的工具层。其核心作用是将上层 AIMCP 请求转化为调度运行时中的任务请求，例如：

(1) 反应任务请求转为反应类调度任务。

(2) 测量任务请求转为测量类调度任务。

(3) 表征任务请求转为表征类调度任务。

(4) 调度状态查询请求直接返回运行时状态。

(5) 调度推进请求驱动任务继续执行。

### 4.5 Agent 模块实现

当前 Agent 模块主要完成以下工作。

(1) 启动 BO Server 与 Action Server。

(2) 发现所有可用工具。

(3) 将演示任务发送给 Action Server。

(4) 获取调度返回结果并显示运行状态。

在中期阶段，Agent 已经能够承担系统顶层入口的角色，为后续引入大模型自动决策机制预留了接口。


## 5. 测试分析与案例分析

### 5.1 测试目标

当前阶段测试的主要目标在于验证以下问题。

(1) AIMCP 与调度模块是否已经打通。

(2) 任务是否能够成功从提交态进入运行态。

(3) 调度运行时是否能够推进任务直至完成。

(4) 资源状态是否能够在任务完成后正确释放。

### 5.2 测试方法

当前阶段主要采用本地演示测试方法，即在集成版本目录下运行：

```powershell
python agent.py
```

系统将自动执行一个最小闭环流程，包括提交反应任务、提交测量任务、查询状态并推进至完成。

### 5.3 测试结果分析

测试中观察到如下现象。

(1) Agent 能够成功启动 BO Server 与 Action Server。

(2) Agent 能够成功发现两个服务端所暴露的工具。

(3) 反应任务与测量任务均能被 Action Server 接收并提交给调度运行时。

(4) 任务状态能够从 `RUNNING` 推进到 `COMPLETED`。

(5) 工作站和机器人在任务完成后能够恢复为空闲状态。

这说明当前系统已经能够完成最小闭环验证。

### 5.4 案例分析

本阶段典型案例为一个简单演示流程。

(1) Agent 首先发起一个 `robotic_reaction` 请求，提交反应任务。

(2) 调度器为该任务分配反应工作站，并将任务状态置为运行中。

(3) 随后 Agent 发起一个 `robotic_measurement` 请求，提交测量任务。

(4) 调度器为该任务分配测量工作站，并记录相应运行状态。

(5) 最后通过 `scheduler_run_until_complete` 推进调度器运行。

(6) 系统返回任务完成事件，并显示所有工作站最终回到 `IDLE` 状态。

该案例表明，目前系统已经具备展示“任务提交 - 资源调度 - 状态推进 - 任务完成”这一核心链路的能力。


## 6. 当前存在的问题

从中期阶段看，当前系统虽然已经具备初步结构和运行能力，但仍存在以下问题。

(1) 当前执行过程仍以本地模拟为主，尚未接入真实实验设备。

(2) 多步复杂实验流程尚未形成稳定的完整链路。

(3) 任务完成主要依赖时间推进，回执闭环机制尚未建立。

(4) LLM 自动规划与调度协同的完整能力仍需继续打磨。


## 7. 下一阶段计划

结合当前中期进展，下一阶段主要计划如下。

(1) 扩展多步实验流程，使反应、测量和表征形成完整任务链。

(2) 增加调度器与 Agent 之间的反馈信息，使 Agent 能根据调度结果继续决策。

(3) 引入更接近真实系统的回执与异常处理机制。

(4) 为后续中期展示和课程汇报继续完善测试结果与文档材料。


## 8. 结论

总体而言，本项目在中期阶段已经完成了从概念分析到原型集成的重要过渡。系统已具备较为清晰的模块划分、初步可运行的调度闭环，以及较明确的后续扩展方向。后续若能进一步补齐复杂流程、回执机制和真实执行接口，则有望形成一个较为完整的自动化实验任务调度原型系统。
